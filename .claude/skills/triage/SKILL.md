# triage

## Single sources

- Label scheme · 3-axis gate · state transitions → `docs/agents/triage-labels.md`
- Linear operations (issues, labels, comments, states) → `docs/agents/issue-tracker.md`
- Writing agent briefs → [AGENT-BRIEF.md](AGENT-BRIEF.md)

## Arguments

- `[issue-identifier]`: optional. Given `CLA-123`, judge only that issue. Otherwise walk the whole un-triaged queue.

## Disclaimer line

Every comment this skill posts starts with this line.

```
> *This was generated by AI during triage.*
```

## Workflow

### 1. Collect the queue (no argument)

Query `list_issues` for the Claude Hunt team (UUID and `includeArchived: false` — see `docs/agents/issue-tracker.md`) and merge two sets:

- issues labeled `needs-triage`
- issues carrying none of the triage state labels (`needs-triage` · `needs-info` · `ai-ready` · `needs-human` · `wontfix`)

Keep only issues whose state is Backlog or Todo. If the queue is empty, print one line — "Nothing to triage" — and stop.

### 2. Gather context per issue

- Read the body, labels, and state with `get_issue`; read the discussion with `list_comments`.
- Inspect the code area the issue points at in the repository. CLAUDE.md (FSD architecture, Testing) is the domain reference.
- Do not reproduce or run anything. Read only as much as the judgment requires.

### 3. Judge

- Pick one **Type label** (`bug` · `feat` · `refactor` · `chore` · `docs`).
- Run the **3-axis gate**. Axis criteria and verdict rules are in the "3-axis gate" section of `docs/agents/triage-labels.md`.

### 4. Apply the verdict

Change labels with `save_issue`. The `labels` parameter overwrites the full set — include the existing labels.

- **`ai-ready`**: post a brief comment using the [AGENT-BRIEF.md](AGENT-BRIEF.md) template, then apply Type + `ai-ready`.
- **`needs-info`**: comment specifically which axis is waiting for which one line, then apply Type + `needs-info`. Vague asks like "please add more detail" are forbidden.
- **`needs-human`**: comment which axis failed and why, then apply Type + `needs-human`.
- **`wontfix`**: comment the reason politely, apply Type + `wontfix`, and move `state: "Canceled"`.

### 5. Summarize

When done, output one table.

```
| Issue | Type | Verdict | Note |
|---|---|---|---|
| CLA-301 | docs | ai-ready | brief posted |
| CLA-302 | feat | needs-human | decision axis failed: design judgment remains |
```

## Quick state changes

When the user says "move CLA-123 to ai-ready", skip the judgment and apply it as-is. Confirm the changes to apply (labels, comment, state) in one line and execute. When moving an issue without a brief to `ai-ready`, ask whether to write one.

## Out of scope

- **No bug reproduction.** Reproducing and running is dispatch's job, when it works the issue.
- **No needs-info round-trip tracking.** When the author replies, a human flips the label back to `needs-triage`.
- **No rejection knowledge base.** The wontfix reason in the issue comment is the entire record.
- Creating issues and writing PRDs are not this skill's job — `capture-issue` owns intake.
