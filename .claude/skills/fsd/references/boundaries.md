# Boundaries — Import Rules and Public API

## Import direction

Strictly one-way, bottom-up:

```
shared   ─┐
entities ─┼→ features ─→ widgets ─→ core ─→ app
          │
          └── (entities may import shared; features may import entities + shared; etc.)
```

- Higher layers import lower. **Never** the reverse.
- `shared` imports nothing from the project (only third-party).

## Slice isolation (same-layer ban)

On the same layer, slices cannot import each other:

- `features/submit-project` **cannot** import from `features/toggle-vote`.
- `entities/project` **cannot** import from `entities/vote`.
- `widgets/header` **cannot** import from `widgets/project-grid`.

If two slices need the same code, promote it:
- Shared between two features → push to the relevant entity (if domain-bound) or to shared (if not).
- Shared between two entities → push to shared.

### Why

This makes slice deletion safe. Delete `features/toggle-vote/` and nothing else breaks, because nothing on the same layer depended on it.

## Public API — `index.ts`

Every slice exposes its surface through `index.ts`. External code imports from the slice barrel:

```ts
// GOOD
import { LoginForm, signOut } from "@features/auth-login"

// BAD — reaches into internal path
import { LoginForm } from "@features/auth-login/ui/login-form"
```

### What to export

From `features/<slice>/index.ts`:

```ts
export { LoginForm } from "./ui/login-form"
export { signOut } from "./api/actions"
```

Re-export only what external consumers need. Internal helpers stay inside.

### Slice-internal tests

Tests living **inside** a slice use **relative imports** — not the barrel:

```ts
// features/auth-login/ui/login-form.test.tsx
import { LoginForm } from "./login-form"
```

This avoids pulling the entire barrel (including server-side modules) into a unit test, which trips env validation in test environments.

External tests (like page-level integration tests) use the barrel:

```ts
// app/__tests__/home.test.tsx
vi.mock("@features/auth-login", () => ({
  signOut: vi.fn(),
}))
```

## Database / infrastructure types

DB row types generated by an external tool (e.g. Supabase) live in `shared/api/<service>/types.ts`. They are **not** FSD domain types. Usage rule:

1. `shared/api/<service>/types.ts` — the generated file. Don't hand-edit.
2. `entities/<domain>/model/schema.ts` — imports the raw type and narrows it to a domain type (`Profile`, `Project`).
3. Feature/page code imports domain types from `@entities/<domain>`.

Biome rule: block direct imports of `@shared/api/<service>/types` outside `shared/api/**` and `entities/**/model/**`. Feature code that reaches through this boundary will fail lint.

## Cross-slice escape hatches (use sparingly)

If you genuinely need to cross a slice boundary without promotion, the FSD community has the `@x` notation:

```
entities/song/@x/artist.ts    ← explicit one-way reference from song to artist
```

Prefer promotion over `@x`. Use `@x` only when the dependency is essential and unidirectional.

## Enforcement gaps

- **Biome `noRestrictedImports` is global** (can't express from/to). It can block imports of a specific path, but it can't say "from features/A, block features/B". Feature→feature imports rely on code review.
- **`supabase.from("table")` leaks typed rows** without importing `Database` directly. The DB-type boundary rule is a speed bump, not a wall.

Dedicated FSD linters (`@feature-sliced/steiger`) or `dependency-cruiser` can enforce these rules if needed, at the cost of extra tooling.
