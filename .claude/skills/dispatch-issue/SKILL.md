# dispatch-issue

## Single sources

- Label scheme · state transitions → `docs/agents/triage-labels.md`
- Linear operations (issues, labels, comments, states) · branch naming → `docs/agents/issue-tracker.md`
- Interpreting agent briefs → `.claude/skills/triage/AGENT-BRIEF.md`
- Code-change workflow → CLAUDE.md ("Testing" principle, "Architecture — FSD" rules)

## Arguments

- `[issue-identifier]`: optional. Given `CLA-123`, run **single mode** and work that issue in this session. Otherwise run **queue mode** ([Queue walk](#queue-walk-queue-mode)).

## Preconditions

- Run from the main checkout or a single worktree. `EnterWorktree` fails inside a nested worktree.
- `gh` CLI is logged in.

## Disclaimer line

Every comment this skill posts starts with this line.

```
> *This was generated by AI during dispatch.*
```

## Workflow (single mode)

### 1. Gather context

- Read the body, labels, and state with `get_issue`; read the agent brief from `list_comments`.
- If there is no brief comment, go to [Failure handling](#failure-handling) (reason: triage incomplete, no brief).
- If the state is already In Progress, another session owns it. Report "already in progress" and stop.

### 2. Mark the pickup

`save_issue` with `state: "In Progress"`. From this point no other dispatch run picks this issue up.

### 3. Branch into a worktree

Create a worktree named `dispatch-cla-<N>` (`CLA-123` → `dispatch-cla-123`).

- Session inside this repo: use `EnterWorktree`.
- Headless or outside-repo session: replicate the hook directly from the repo root —
  `echo '{"name":"dispatch-cla-<N>"}' | bash .claude/hooks/worktree-create.sh`
  (rebases main onto origin, copies gitignored `.env*` files, runs `bun install`).

The Linear GitHub integration detects `cla-<n>` in the branch name and auto-links the issue. Do not use the `gitBranchName` Linear suggests — it can contain Korean characters.

### 4. Work

The brief is the contract. Satisfy the acceptance criteria one by one and leave everything out of scope untouched.

- **Code changes** (component behavior, utils, route handlers, server actions): follow the CLAUDE.md Testing principle — define success criteria, write the test that proves each criterion, loop until verified. Respect FSD layer and slice-isolation rules.
- **Docs and config** (`docs/`, rules, skills, CLAUDE.md, settings): not test-driven. Verify against the brief's acceptance criteria.

**Hard aborts** — if the work turns out to require any of the following, the `ai-ready` verdict was wrong; go to [Failure handling](#failure-handling) instead of doing it:

- A new external dependency (`package.json` deps, `bun add`)
- Environment changes (`.env*`, new env-var reads, deploy-affecting config)
- DB schema / RLS / migrations (`supabase/schemas/`, `supabase/migrations/`, policy SQL)
- Edits beyond the issue's scope — no drive-by refactors or adjacent cleanups

### 5. Validate

Run in order and stop at the first failure (its stage name goes in the failure comment):

1. `bun run typecheck`
2. `bun run check`
3. `bun run test:unit`
4. `bun run build`

`test:db` and `test:e2e` are intentionally excluded — they need a running Supabase instance the worktree does not have.

### 6. Commit

- Message: `<type>(<scope>): CLA-<N> <short summary>`. `<type>` comes from the issue's Type label (`bug` → `fix`), `<scope>` from the changed area.
- One logical change = one commit.

### 7. Push + create the PR

```bash
git push -u origin dispatch-cla-<N>
gh pr create --title "<type>(<scope>): CLA-<N> <summary>" --body "$(cat <<'EOF'
Linear: <issue URL>

## Summary

<brief Summary and how each acceptance criterion was satisfied>

## Validation

- [x] bun run typecheck
- [x] bun run check
- [x] bun run test:unit
- [x] bun run build

Automated via `/dispatch-issue`.
EOF
)"
```

### 8. Update the issue

- `save_comment` with the PR URL (disclaimer line first).
- `save_issue` with `state: "In Review"`.

### 9. Return

Call `ExitWorktree` with `action: "keep"` (skip when the worktree was created via the hook fallback). Output: the PR URL, one line.

## Failure handling

When blocked at any step:

1. `save_issue`: add the `dispatch-failed` label (full list including existing labels) and move `state: "Todo"`.
2. `save_comment`: which step was blocked and why (disclaimer line first).
3. Leave the worktree and branch in place for diagnosis.
4. Stop. A human retries by removing `dispatch-failed` and running `/dispatch-issue CLA-<N>`.

## Queue walk (queue mode)

Called with no argument, this session is the orchestrator. It does no issue work itself; it delegates each issue to an independent agent so context never bleeds between issues.

1. `list_issues` for the Claude Hunt team (UUID, `includeArchived: false`) with `label: "ai-ready"`; keep issues in Backlog or Todo without the `dispatch-failed` label, oldest first. If the queue is empty, print "Nothing to dispatch" and stop.
2. For each issue, spawn a fresh agent (Agent tool) that runs this skill in single mode (`CLA-<N>`). One issue per agent, fresh context every time.
3. Run sequentially. One agent must finish before the next starts. Parallel runs are forbidden — the worktree-create hook's main rebase races.
4. The orchestrator never reads issue bodies or diffs. It records only each agent's report (PR URL or failure reason). One failure does not block the next issue.
5. Summarize as a table.

```
| Issue | Result | PR / Reason |
|---|---|---|
| CLA-301 | ✅ | https://github.com/.../pull/41 |
| CLA-302 | ❌ dispatch-failed | no brief |
```

## Out of scope

- **Humans merge.** This skill ends at PR creation.
- PR review comments and follow-up fixes are not handled here.
- Triage (classification, brief writing) belongs to the `triage` skill. On an issue without a brief, fail — do not classify it yourself.
- Worktree cleanup happens after the human merges (`git worktree remove .claude/worktrees/dispatch-cla-<N>`).
