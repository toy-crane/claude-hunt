# UX 라이팅 원칙

claude-hunt의 모든 사용자 노출 문구(버튼, 토스트, 에러, 빈 상태, 폼, 다이얼로그, 메타데이터)에 적용한다. 토스 프로덕트 라이팅의 코어밸류(명확·간결·친근·존중·공감)를 우리 서비스에 맞게 옮긴 것이다.

## 보이스

같은 수업을 듣는 동료의 목소리. 친근하게 응원하되 호들갑 떨지 않는다. 금융 서비스가 아니므로 격식보다 가벼움이 기본값이지만, 삭제·탈퇴 같은 파괴적 행동 앞에서는 정확하고 정직하게 말한다.

## 원칙

1. **다음 화면을 예고한다** — 버튼·링크 라벨만 읽고도 누른 뒤의 화면을 예상할 수 있어야 한다. 막연한 "확인" 대신 "삭제하기", "제출하기".
2. **잡초를 뽑는다** — 빼도 뜻이 달라지지 않는 단어·문장은 뺀다. 제목이 이미 말한 내용을 본문에서 반복하지 않는다.
3. **말하듯 쓴다** — 소리 내어 읽었을 때 자연스러워야 한다. 한자어·문어체·개발 용어(그리드, 유효하지 않은, 세션, 인증)를 일상어로 바꾼다.
4. **능동형으로 쓴다** — "제출되었어요" → "제출했어요". 시스템이 주체일 때("서비스가 종료돼요")나 사용자를 안심시킬 때("안전하게 보관돼요")만 수동형 허용.
5. **되는 길을 알려준다** — 안 되는 이유보다 지금 할 수 있는 행동을 먼저. 에러 메시지는 "무엇이 안 됐는지 + 어떻게 하면 되는지" 구조.
6. **강요하지 않는다** — 공포·손해 프레임 금지. 선택권이 사용자에게 있음을 느끼게 한다.
7. **숨은 감정에 공감한다** — 첫 제출, 이달의 1위, 빈 목록 앞에서 사용자가 느낄 감정을 한 번 생각하고 쓴다. 단, 과한 이모지·호들갑은 잡초다.

## 문장 규칙

| 항목 | 규칙 | 예 |
|------|------|----|
| 문체 | 모든 문구는 해요체. 합니다체 금지 | "삭제됩니다" → "삭제돼요" |
| 성공 토스트 | "~했어요." (능동형 + 마침표) | "프로젝트를 제출했어요." |
| 실패 토스트 | "~하지 못했어요." 뒤에 다음 행동 | "댓글을 등록하지 못했어요. 잠시 후 다시 시도해 주세요." |
| 검증 메시지 | 필수 입력: "~을/를 입력해 주세요." · 글자 수 한도: "~자까지 쓸 수 있어요." | |
| 확인 다이얼로그 | 제목은 "~할까요?", 본문은 결과를 정직하게, 확인 버튼은 동사형 | "프로젝트를 삭제할까요?" / [삭제하기] |
| 닫기 vs 취소 | 진행 중인 행동을 멈추면 "취소", 정보만 닫으면 "닫기" | |
| 표기 | "되어요" → "돼요", "해주세요" → "해 주세요"로 통일 | |
| 조사 | 받침에 맞는 조사를 쓴다. 동적 값 뒤에는 "을(를)" 병기 대신 문장 재구성 우선 | "한 줄 소개을" → "한 줄 소개를" |
| 용어 | 같은 개념은 같은 단어로. 추천(투표 아님), 프로젝트, 댓글, 클래스 | |
| 언어 | 사용자에게 보이는 문구는 모두 한국어. 고유명사(GitHub, Claude Code)는 예외 | |

## 적용 범위

- 적용: JSX 텍스트, 토스트, zod 검증 메시지, placeholder, aria-label, 다이얼로그, 빈 상태, `metadata` title/description, OG 이미지 텍스트
- 제외: 개발자용 로그·내부 throw 메시지, 코드 식별자
- 문구를 바꾸면 그 문구를 단언하는 테스트(`*.test.tsx`, `e2e/*.spec.ts`)도 같은 커밋에서 함께 바꾼다
