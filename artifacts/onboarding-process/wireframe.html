Notes

Entry to app for any signed-in user whose cohort_id is null. Hard gate: any authed page redirects here.

  • Trigger: signed-in user with cohort_id IS NULL hits any authed route → redirect to /onboarding?next=<original> (scenario 1, 6)
  • Interaction: type display name, pick cohort from dropdown, press Continue
  • Transition (success): Continue → profile updated → redirect to next (or /) (scenario 2)
  • Transition (invalid): stays on screen with inline errors → see "Validation errors" screen (scenario 3)
  • Transition (escape): Sign out → session cleared → redirect to /login
  • Not shown here: onboarded user visiting /onboarding is redirected away (scenario 4) — no UI

Other error variants shown on the same screen (not stacked):
  • Whitespace-only name → "Display name is required"
  • > 50 chars → "Display name must be 50 characters or fewer"
Notes

Same layout as the base onboarding form. Only the inline error states differ.

  • Trigger: user presses Continue with missing or invalid input (scenario 3)
  • Interaction: error messages appear under each invalid field; valid values already entered are preserved
  • Transition: user corrects the fields and presses Continue again → goes to the "Onboarding form" happy path
  • Scope: all three validation cases (empty, whitespace-only, over-length) map to this screen

Notes

Degraded state when the cohorts table is empty. The selector is hidden (not just disabled) so the user is not misled into trying it.

  • Trigger: signed-in user hits onboarding while zero cohort rows exist (scenario 5)
  • Interaction: Continue is disabled — the user cannot submit an incomplete profile; Sign out remains available so they are not trapped
  • Transition: once a cohort is seeded by the instructor, the next visit shows the base "Onboarding form" screen
  • Rationale: per spec invariant "Completion condition is single-field" — without a cohort there is nothing to save that would clear the gate
