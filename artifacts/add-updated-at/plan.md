# Add updated_at — Implementation Plan

## Context

Four `public.*` tables violate `.claude/rules/supabase-timestamps.md`: `profiles` and `projects` have the `updated_at` column but are missing the `moddatetime` trigger, and `cohorts` and `votes` lack both. The rule requires every `public.*` table to expose `updated_at` as the last column and advance it automatically on every update. This plan brings all four tables into compliance while preserving existing data and RLS.

## Architecture Decisions

| Decision | Choice | Reason |
|----------|--------|--------|
| Column default on existing tables | `timestamptz default now() not null` | Matches the rule's exact template; existing rows receive `now()` at migration, satisfying spec scenario 4 (`updated_at ≥ created_at`, not in future). |
| Backfill of pre-existing rows | None explicit — rely on `default now()` | Satisfies spec invariants without extra DML migration. Cohorts are seeded via migration; votes/profiles/projects hold live data but `now() ≥ created_at` holds trivially. |
| Migration granularity | One trigger migration per table (`add_<table>_updated_at_trigger`) | Matches the rule's literal template; smaller, individually revertable blast radius. |
| Trigger extension | `moddatetime` from `extensions` schema | Exactly as prescribed by the rule; `create extension if not exists` is idempotent. |
| Test strategy | pgTAP only, extend the existing `supabase/tests/<table>_test.sql` file | Project testing convention; `bun run test:db` already wired. No Vitest/E2E value for a trigger. |

## Infrastructure Resources

None. No new buckets, cron jobs, or env vars. The `moddatetime` extension ships with Postgres and is enabled idempotently in each trigger migration.
