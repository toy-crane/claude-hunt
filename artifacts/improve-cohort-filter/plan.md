# Improve Cohort Filter — Implementation Plan

## Architecture Decisions

| Decision | Choice | Reason |
|----------|--------|--------|
| Loading feedback mechanism | Server Component `<Suspense key={cohortId}>` wrapping the data-fetching child | Standard Next.js App Router pattern; keying on `cohortId` forces re-suspend on filter change so the skeleton always shows, no client-state library needed |
| Where to source `vote_count` | Denormalized `projects.vote_count` column maintained by a trigger on `votes` INSERT/DELETE | Textbook counter-cache; removes the full-table `GROUP BY votes` scan from the view; reads become O(rows returned) instead of O(total votes) |
| Definer-mode view | Keep `projects_with_vote_count` with `security_invoker = false` untouched | Supabase's recommended pattern for PII-safe anon access; perf fix is orthogonal to security model (see spec Excluded list) |
| Viewer-votes index | Rely on existing composite `(user_id, project_id)` unique index; add no new index | Postgres uses the leading column of a composite index for `user_id`-only lookups; a standalone `votes(user_id)` index would duplicate coverage |
| Cohorts list caching | `react.cache()` wrapper on `fetchCohorts` (absorbed into Task 1 as a one-line hardening) | Built-in Next.js primitive; cheap insurance against future duplicate callers; not worth a dedicated task or mock-based acceptance criterion |
| Skeleton primitive | shadcn `skeleton` component installed via its registry | Project rule mandates shadcn for UI primitives; no custom markup |

## Data Model

### Project (existing)
- id, user_id, cohort_id, title, tagline, project_url, screenshot_path, created_at, updated_at
- **`vote_count` (new)** — non-negative integer, maintained by a trigger on `votes`; initial backfill from existing rows

### Vote (existing, unchanged shape)
- id, user_id, project_id, created_at, updated_at
- Trigger: on INSERT → `projects.vote_count += 1`; on DELETE → `projects.vote_count -= 1`

### `projects_with_vote_count` view (simplified)
- Same output columns as today
- Internally reads `p.vote_count` directly; no sub-aggregate over `votes`
- `security_invoker = false` preserved; only display-safe author columns exposed

### Task 2: Project vote counts are served from a denormalized column maintained by a trigger

- **Covers**: Scenario 2 (full), Scenario 3 (full), Scenario 4 (full), Invariant "vote counts match authoritative state", Invariant "anonymous PII boundary"
- **Size**: M (5 files + test files)
- **Dependencies**: None for runtime; recommend landing after Task 1 so the skeleton already masks any remaining latency
- **References**:
  - supabase — declarative path for column + view changes; Manual Migration Path for trigger and DML backfill
  - supabase-postgres-best-practices — counter-cache pattern; idempotent trigger
  - security-and-hardening — negative-path pgTAP for anon PII boundary
  - test-driven-development — pgTAP test first
  - `.claude/rules/supabase-migration.md` (project rule: never edit migration files by hand for declarative changes)
  - `.claude/rules/supabase-timestamps.md` (precedent for manual-path trigger migrations using `extensions.<fn>`)
  - `supabase/schemas/projects.sql`, `supabase/schemas/projects_with_vote_count.sql`, `supabase/schemas/votes.sql`
- **Implementation targets**:
  - `supabase/schemas/projects.sql` (add `vote_count bigint not null default 0`; `created_at` / `updated_at` must stay as the last two columns per project rule)
  - `supabase/schemas/projects_with_vote_count.sql` (replace sub-aggregate with `p.vote_count`)
  - Generated: `supabase/migrations/<ts>_add_vote_count_to_projects.sql` via `supabase db diff -f add_vote_count_to_projects`
  - Generated: `supabase/migrations/<ts>_simplify_vote_count_view.sql` via `supabase db diff -f simplify_vote_count_view`
  - Manual: `supabase migration new maintain_projects_vote_count` — `CREATE FUNCTION public.maintain_project_vote_count()` + `CREATE TRIGGER` on `public.votes` for `AFTER INSERT OR DELETE FOR EACH ROW`
  - Manual: `supabase migration new backfill_projects_vote_count` — `UPDATE public.projects p SET vote_count = (SELECT count(*) FROM public.votes v WHERE v.project_id = p.id)`
  - `supabase/tests/projects_vote_count_trigger_test.sql` (pgTAP — increment, decrement, never-negative)
  - `supabase/tests/projects_with_vote_count_security_test.sql` (pgTAP — as `anon`, select list from the view excludes any PII column; direct select on `profiles.email` as `anon` is denied)
  - `supabase/tests/projects_test.sql` (add `has_column('public','projects','vote_count',...)`)
  - `e2e/project-board.spec.ts` (extend the authed-student scenario: after voting on two projects in the selected cohort, reload and assert exactly those two cards show the voted indicator)
- **Acceptance**:
  - [ ] Inserting a row into `votes` for project P increments `projects.vote_count` for P by exactly 1
  - [ ] Deleting a row from `votes` for project P decrements `projects.vote_count` for P by exactly 1; `vote_count` never goes negative in normal flows
  - [ ] Querying `projects_with_vote_count` for any cohort returns the same ordering and counts as the current implementation, and the "All cohorts" project count equals the sum of per-cohort counts
  - [ ] After backfill, every row's `projects.vote_count` equals `count(*) from votes where project_id = projects.id`
  - [ ] As the `anon` Postgres role, selecting from `projects_with_vote_count` returns the author display name and avatar but never `profiles.email` or other PII columns; a direct `select email from public.profiles` as `anon` is denied
  - [ ] For a logged-in user who has voted on two specific projects in the selected cohort, exactly those two cards show the "voted" indicator after the grid renders (E2E)
- **Verification**:
  - `supabase db reset` (applies all migrations cleanly)
  - `bun run test:db` (pgTAP — trigger, security, column presence)
  - `bun run test:e2e -- project-board`
