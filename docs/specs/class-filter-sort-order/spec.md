# 클래스 필터 정렬 순서

클래스 필터 목록을 최신 기수가 맨 위로 오도록 바꾼다. 기수가 아닌 클래스(인프런, toycrane)는 양 끝에 고정한다.

## 배경

지금 `fetchCohorts`는 `name` 오름차순으로 정렬한다. 프로덕션 기준 결과는 `LG전자 1기 → 7기 → toycrane` 이고, 가장 오래된 기수가 맨 앞이다. 최신 기수를 쓰는 사람이 제일 많은데 목록 끝까지 내려가야 한다.

단순히 내림차순으로 뒤집으면 `T > L` 이라 **toycrane이 맨 위로 올라온다**. 기수가 아닌 항목이 최상단을 차지하는 게 이 작업의 출발점이었다.

## 결정

### 1. 정렬 키는 `created_at`. 틀어진 행 2개만 고친다

새 칼럼을 두지 않는다. 클래스가 열릴 때 행이 쓰이므로 `created_at` 내림차순이 곧 최신순이고, 앞으로 추가되는 클래스는 아무 손질 없이 제자리에 들어간다.

프로덕션에서 순서가 어긋난 행은 **둘뿐이다**:

| name | 현재 값 | 문제 | 고칠 값 |
|------|---------|------|---------|
| `LGE-1` | 2026-05-11 | LGE-2·3보다 한 달 뒤에 쓰여서 1기가 3기 위로 올라옴 | 2026-04-14 06:00:50 |
| `LGE-3` | 2026-04-14 06:00:52.821638 | LGE-2와 소수점까지 동일 — 순서 비결정적 | 2026-04-14 06:00:54 |

나머지 5행은 이미 맞게 정렬되므로 건드리지 않는다. `cohorts.created_at`을 읽는 코드가 없어서 이 UPDATE에 걸릴 소비자도 없다.

### 2. seed는 `created_at`을 명시적으로 적는다

`now()`는 **트랜잭션** 타임스탬프다. 한 문장이든 여러 문장이든 같은 트랜잭션 안에서는 모든 행이 같은 값을 받는다(로컬에서 확인함). seed의 클래스 5개를 기본값에 맡기면 전부 동점이 되고, 이름 보조 키가 사전순으로 정렬해 `LGE-1`이 맨 위로 온다 — 지금 고치려는 바로 그 버그다.

프로덕션의 `LGE-2`/`LGE-3` 동점도 정확히 이 원인이다. 두 행이 한 문장에서 들어갔다는 증거다.

### 3. 인프런과 toycrane 고정은 코드가 한다

양 끝 고정은 이름 기반 규칙으로 `entities/cohort`에 둔다.

```
PINNED_FIRST = ["Inflearn"]    → 항상 맨 위
PINNED_LAST  = ["TOYCRANE"]    → 항상 맨 끝
그 외         → created_at 내림차순, 동점이면 name
```

이 코드베이스엔 이미 똑같은 모양의 규칙이 있다 — `entities/cohort/model/selectable.ts`의 `NON_SELECTABLE_COHORT_NAMES = ["TOYCRANE"]`. 예외를 이름으로 적어두는 방식이 낯설지 않고, 바로 옆에 붙는다.

**날짜로 표현할 수 없기 때문이다.** "항상 맨 위"는 날짜가 아니다. 흉내 내려면 아무도 참으로 읽을 수 없는 미래 시각을 골라야 하고, 더 새로운 클래스가 생길 때마다 다시 골라야 한다.

### 4. 비교는 문자열이 아니라 인스턴트로

PostgREST가 timestamptz를 오프셋과 함께 내려주는데, 같은 시각의 두 표기(`…Z`와 `…+00:00`)는 사전순으로 정렬되지 않는다. `new Date(...)`로 파싱해서 비교한다.

### 4. 세 화면에 모두 적용한다

`fetchCohorts` 하나를 고치면 보드 필터(칩 레일 + 콤보박스), 온보딩 클래스 선택, 설정 클래스 선택이 함께 바뀐다. 화면별로 다른 순서를 쓸 이유가 없다. 신규 가입자는 최신 기수일 확률이 높으니 온보딩에서도 최신순이 맞다.

### 5. `isSelectableCohort`는 그대로 둔다

toycrane을 온보딩·설정 선택지에서 빼는 규칙은 **선택** 가능 여부이고, 정렬은 **표시** 순서다. 서로 다른 관심사이고 지금 분리돼 있는 상태가 맞다.

## 근거 — 다른 방법을 쓰지 않는 이유

**전용 정렬 칼럼을 두지 않는다.** 처음엔 `display_order int generated by default as identity`로 구현했다가 되돌렸다(`2d355d7`). 동작은 했지만 값을 바로잡아야 하는 행이 **2개뿐**인데 칼럼 하나, 마이그레이션 2개, 타입 재생성, 픽스처 6파일을 들이는 건 과했다. `cohorts.created_at`을 읽는 코드가 없어서 값을 고쳐도 깨질 소비자가 없다는 걸 확인한 게 결정적이었다.

되돌리면서 잃은 것도 있다. identity는 한 문장에 여러 행을 넣어도 값이 갈리지만 `created_at`은 그렇지 않다 — 그래서 seed에 명시적 값이 필요해졌고, "클래스를 배치로 넣으면 동점"이라는 함정이 남는다. 결정 3의 이름 고정 규칙은 두 방식 모두에 필요했으므로 그대로다.

**`name` 파싱은 못 쓴다.** [add-seed-sql/spec.md:56](../../../artifacts/add-seed-sql/spec.md)이 슬러그를 "테스트·URL·서버 조회가 쓰는 안정적 식별자"로 못 박았다. 표시 순서를 여기서 유추하면 그 계약을 넘어선다. 실무적으로도 `Inflearn`엔 숫자가 없어서 답이 안 나오고, `LGE-10`이 생기면 사전순으로 `LGE-1`과 `LGE-2` 사이에 낀다.

**`program` + `sequence`로 나누지 않는다.** 인프런이 기수별로 늘어난다면 프로그램 우선순위와 기수 번호를 분리하는 게 맞지만, 인프런은 기수 구분 없이 한 항목으로 들어온다. 이름 고정 두 줄로 충분하다.

**프로젝트 활동량 순으로 정렬하지 않는다.** 프로젝트가 없는 신규 기수가 맨 아래로 가서 목적과 정반대가 된다.

## 선행 작업 — PR #180 (닫힘)

이 작업은 한 번 만들어졌다가 버려졌다. 브랜치 `cohort-filter-ui-0a7f76`, [PR #180](https://github.com/toy-crane/claude-hunt/pull/180), 커밋 `db35851` + `31abe8c`.

그 구현은 `display_order int not null default 0` 칼럼을 두고 이름에서 값을 백필했다. 버려진 이유는 정렬이 아니다. PR이 **칩 행을 드롭다운으로 교체하는 UI 변경과 묶여 있었고**, main은 그 뒤 다른 방향(콤보박스 + 한 줄 레일 병행)으로 갔다. UI가 반려되면서 정렬까지 같이 사라졌다.

**가져오지 말 것**: `3c364eb`, `4aa39ab` (UI 커밋 — 이미 대체됨).

그 구현을 그대로 되살리지 않는 이유는 둘이다:

1. **`default 0`** — 클래스를 추가할 때 숫자를 매번 적어야 하고, 빠뜨리면 0이 들어가 조용히 맨 아래로 간다. 게다가 `default 0`이 toycrane을 맨 끝에 두는 장치로도 쓰이고 있어서(주석에 그렇게 적혀 있다), 값을 빠뜨린 새 클래스가 toycrane과 같은 자리를 두고 경쟁한다.
2. **인프런 고정이 없다** — seed에서 인프런에 4를 줘서 우연히 맨 위일 뿐이다. `LGE-5`가 들어오는 순간 5 > 4라서 밀려난다. 고정이 아니라 그냥 숫자다.

지금 방식은 칼럼 자체를 없애고 고정을 이름 규칙으로 옮겨 두 문제를 모두 피한다.

## 검증 결과 — 실제 데이터

프로덕션(8행)에 제안한 규칙을 그대로 돌린 결과:

```
LG전자 7기(0) → 6기(20) → 5기(22) → 4기(13) → 3기(20) → 2기(9) → 1기(3) → toycrane(3)
```

괄호는 프로젝트 수다. `LGE-1`과 `LGE-3` 두 행만 고친 상태로 시뮬레이션한 결과이며, 목표한 순서와 정확히 일치한다. 프로젝트 90개 전부 클래스에 연결돼 있고 고아 행이 없다.

인프런 고정은 로컬 seed로만 확인 가능하다(프로덕션에 인프런 행이 없다). `인프런 → 3기 → 2기 → 1기 → toycrane` 순으로 양쪽 고정이 모두 걸린다.

## 가정 — 다르면 알려주세요

- **"모든 클래스"는 계속 맨 앞 고정.** 정렬을 아예 거치지 않는, 목록 밖의 항목이다.
- **프로젝트 0개인 클래스도 그대로 보여준다.** 최신인 7기가 지금 0개라 맨 앞 칩이 `LG전자 7기 0`이 된다. 숨기면 정작 자기 클래스를 찾는 7기 수강생에게 안 보이고, 첫 제출이 들어오는 순간 칩이 튀어나와 목록이 흔들린다. 카운트가 이미 빈 상태를 말해주고 있다.
- **기존 필터 동작은 건드리지 않는다.** URL 동기화, 칩 자동 스크롤, 콤보박스 검색은 그대로다.
- **클래스는 한 번에 하나씩 추가한다.** 프로덕션에 새 기수를 넣을 때 `created_at`이 자동으로 붙어 제자리에 들어간다. 여러 개를 한 문장에 넣으면 동점이 되므로 그때는 값을 명시한다.

## 남은 위험

- **`created_at`이 두 행에서 사실과 달라진다.** `LGE-1`은 실제로 2026-05-11에 쓰였는데 04-14로 기록된다. 지금은 읽는 코드가 없지만, "이 클래스 행 언제 만들어졌지"를 나중에 누가 물으면 틀린 답을 얻는다. 표시 순서를 위해 감사 칼럼을 조정한 것이라는 사실이 마이그레이션 주석에 남아 있다.
- **클래스를 배치로 넣으면 조용히 동점이 난다.** `now()`가 트랜잭션 타임스탬프라 한 트랜잭션에서 들어간 행은 INSERT를 쪼개도 값이 같다. 목록이 이상해질 때까지 아무도 모른다 — 전용 정렬 칼럼을 뒀다면 없었을 함정이고, 이 방식을 고른 대가다. 여러 개를 넣을 땐 `created_at`을 명시할 것.
- **프로덕션 마이그레이션과 배포 순서.** `LGE-1`/`LGE-3`이 고쳐지기 전에 새 코드가 뜨면 1기가 3기 위에, 2·3기가 뒤죽박죽으로 보인다. 에러는 안 나므로 더 눈에 안 띈다. 마이그레이션 먼저, 배포 나중.
- **인프런 고정은 프로덕션에서 검증할 수 없다.** 행이 없어서 로컬 seed로만 확인 가능하다. 실제로 들어올 때 맨 위에 오는지 다시 볼 것.
- **고정 목록이 코드에 있다는 건 인프런 추가가 코드 배포를 요구한다는 뜻은 아니다.** `Inflearn` 이름이 이미 목록에 있으므로 행만 INSERT하면 된다. 다만 **다른** 이름으로 들어오면(`inflearn`, `INFLEARN-1` 등) 고정이 안 걸린다. 슬러그를 `Inflearn`으로 맞출 것.
