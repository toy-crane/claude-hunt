# 클래스 필터 정렬 순서

클래스 필터 목록을 최신 기수가 맨 위로 오도록 바꾼다. 기수가 아닌 클래스(인프런, toycrane)는 양 끝에 고정한다.

## 배경

지금 `fetchCohorts`는 `name` 오름차순으로 정렬한다. 프로덕션 기준 결과는 `LG전자 1기 → 7기 → toycrane` 이고, 가장 오래된 기수가 맨 앞이다. 최신 기수를 쓰는 사람이 제일 많은데 목록 끝까지 내려가야 한다.

단순히 내림차순으로 뒤집으면 `T > L` 이라 **toycrane이 맨 위로 올라온다**. 기수가 아닌 항목이 최상단을 차지하는 게 이 작업의 출발점이었다.

## 결정

### 1. `cohorts`에 `display_order` 정수 칼럼을 추가하고, 내림차순으로 정렬한다

정렬 키는 데이터로 명시한다. 이름에서 유추하지 않는다.

| name | label | display_order |
|------|-------|---------------|
| `Inflearn` | 인프런 | 100 |
| `LGE-7` | LG전자 7기 | 7 |
| `LGE-6` | LG전자 6기 | 6 |
| `LGE-5` | LG전자 5기 | 5 |
| `LGE-4` | LG전자 4기 | 4 |
| `LGE-3` | LG전자 3기 | 3 |
| `LGE-2` | LG전자 2기 | 2 |
| `LGE-1` | LG전자 1기 | 1 |
| `TOYCRANE` | toycrane | 0 |

`LGE-N`은 기수 번호를 그대로 쓴다. 새 기수를 넣을 때 적을 숫자가 자명하다.

인프런은 100을 받아 항상 맨 위에 앉는다. toycrane은 0을 받아 항상 맨 끝이다. "양쪽 고정"이 별도 분기 없이 같은 정렬 하나로 표현된다.

프로덕션에는 인프런 행이 없다. 인프런의 100은 seed에만 존재하며, 실제로 들어올 때 이 값을 쓴다.

### 2. `not null`, 기본값 없음

기본값을 두면 새 기수를 넣을 때 숫자를 빼먹어도 INSERT가 통과하고, 그 클래스는 조용히 목록 맨 아래로 간다 — 지금 고치려는 버그와 똑같은 증상이 소리 없이 재발한다. 기본값 없는 `not null`이면 INSERT가 그 자리에서 실패해서 바로 알아챈다.

### 3. `display_order desc, name asc`

동점일 때 순서가 흔들리지 않도록 `name`을 보조 키로 둔다.

### 4. 세 화면에 모두 적용한다

`fetchCohorts` 하나를 고치면 보드 필터(칩 레일 + 콤보박스), 온보딩 클래스 선택, 설정 클래스 선택이 함께 바뀐다. 화면별로 다른 순서를 쓸 이유가 없다. 신규 가입자는 최신 기수일 확률이 높으니 온보딩에서도 최신순이 맞다.

### 5. `isSelectableCohort`는 그대로 둔다

toycrane을 온보딩·설정 선택지에서 빼는 규칙은 **선택** 가능 여부이고, `display_order`는 **표시** 순서다. 서로 다른 관심사이고 지금 분리돼 있는 상태가 맞다.

## 근거 — 다른 정렬 키를 쓰지 않는 이유

**`created_at`은 못 쓴다.** 프로덕션 값을 확인한 결과:

| name | created_at |
|------|-----------|
| LGE-2, LGE-3 | 2026-04-14 (동일 timestamp) |
| LGE-1 | 2026-05-11 |
| LGE-4 | 2026-05-21 |
| LGE-5 | 2026-05-26 |
| TOYCRANE | 2026-07-06 |
| LGE-6 | 2026-07-09 |
| LGE-7 | 2026-07-16 |

1기 행이 2·3기보다 **나중에** 채워졌다. 생성순으로 정렬하면 1기가 3기 위로 올라오고, 2기와 3기는 timestamp가 같아 순서가 비결정적이다. toycrane도 5기와 6기 사이에 끼어든다. 행이 만들어진 시각은 클래스가 열린 순서가 아니다.

**`name` 파싱도 못 쓴다.** [add-seed-sql/spec.md:56](../../../artifacts/add-seed-sql/spec.md)이 슬러그를 "테스트·URL·서버 조회가 쓰는 안정적 식별자"로 못 박았다. 표시 순서를 여기서 유추하면 그 계약을 넘어선다. 실무적으로도 `Inflearn`엔 숫자가 없어서 답이 안 나오고, `LGE-10`이 생기면 사전순으로 `LGE-1`과 `LGE-2` 사이에 낀다.

**`program` + `sequence`로 나누지 않는다.** 인프런이 기수별로 늘어난다면 프로그램 우선순위와 기수 번호를 분리하는 게 맞지만, 인프런은 기수 구분 없이 한 항목으로 들어온다. 칼럼 하나로 충분하다.

**프로젝트 활동량 순으로 정렬하지 않는다.** 프로젝트가 없는 신규 기수가 맨 아래로 가서 목적과 정반대가 된다.

## 영향 범위

| 파일 | 변경 |
|------|------|
| `supabase/schemas/cohorts.sql` | `display_order int not null` 추가 — `created_at`/`updated_at` 앞에 |
| `supabase/migrations/` | 스키마 변경은 `supabase db diff`, 값 백필은 `supabase migration new` (DML은 diff가 못 잡는다) |
| `supabase/seed.sql` | 5개 행에 `display_order` 부여 |
| `shared/api/supabase/types.ts` | 타입 재생성 |
| `features/cohort-filter/api/fetch-cohorts.ts` | `.order("name")` → `.order("display_order", { ascending: false }).order("name")` + 주석 갱신 |
| `features/cohort-filter/api/fetch-cohorts.test.ts` | `order` 호출 단언 갱신 |

## 성공 기준

- [ ] 로컬 seed에서 `fetchCohorts()`가 `인프런 → LG전자 3기 → 2기 → 1기 → toycrane` 순으로 반환한다
- [ ] 보드 콤보박스 목록이 `모든 클래스` 다음부터 같은 순서로 렌더된다
- [ ] 보드 칩 레일이 `모든 클래스` 다음부터 같은 순서로 렌더된다
- [ ] 온보딩 클래스 선택지가 `인프런 → LG전자 3기 → 2기 → 1기` 순이고 toycrane이 없다
- [ ] 설정 클래스 선택지가 같은 순서이고 toycrane이 없다
- [ ] `display_order` 없이 `cohorts`에 INSERT하면 실패한다
- [ ] 프로덕션 마이그레이션 후 목록이 `LG전자 7기 → 6기 → … → 1기 → toycrane` 순이다

## 가정 — 다르면 알려주세요

- **"모든 클래스"는 계속 맨 앞 고정.** `display_order`의 영향을 받지 않는, 목록 밖의 항목이다.
- **프로젝트 0개인 클래스도 그대로 보여준다.** 최신인 7기가 지금 0개라 맨 앞 칩이 `LG전자 7기 0`이 된다. 숨기면 정작 자기 클래스를 찾는 7기 수강생에게 안 보이고, 첫 제출이 들어오는 순간 칩이 튀어나와 목록이 흔들린다. 카운트가 이미 빈 상태를 말해주고 있다.
- **기존 필터 동작은 건드리지 않는다.** URL 동기화, 칩 자동 스크롤, 콤보박스 검색은 그대로다.
- **`display_order`는 사람이 손으로 넣는다.** 클래스는 seed/마이그레이션으로 관리되고 관리 화면이 없다. 새 기수를 추가하는 사람이 이미 SQL을 쓰고 있으므로 숫자 한 칸을 더 적는 부담은 크지 않다.

## 남은 위험

- **`not null` 칼럼을 비어 있지 않은 테이블에 추가하는 순서.** `supabase db diff`는 `add column ... not null`을 그대로 뽑는데, 기존 8개 행 때문에 실패한다. nullable로 추가 → 백필 → `set not null` 3단계로 나눠야 한다. 마이그레이션 리뷰에서 확인할 것.
- **`fetch-cohorts.test.ts`의 목이 `.order()` 체이닝을 못 받는다.** 현재 `select: () => ({ order })`이고 `order`가 프로미스를 반환해서, `.order()`를 두 번 부르면 목이 깨진다. 체이닝 가능하도록 목을 고쳐야 한다.
- **프로덕션 백필과 배포 순서.** 칼럼이 생기기 전에 새 코드가 뜨면 `display_order` 정렬이 에러를 낸다. 마이그레이션 먼저, 배포 나중.
- **`seed.sql`의 낡은 주석.** 21–23행이 "fetchCohorts filters it out"이라고 적혀 있는데 커밋 51557ed 이후로 사실이 아니다. 같은 파일을 건드리는 김에 고칠 것.
- **인프런의 100은 아직 검증되지 않았다.** 프로덕션에 행이 없어서 로컬 seed로만 확인 가능하다. 실제로 인프런이 들어올 때 맨 위에 오는지 다시 볼 것.
