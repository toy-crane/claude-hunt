# 클래스 필터 정렬 순서

클래스 필터 목록을 최신 기수가 맨 위로 오도록 바꾼다. 기수가 아닌 클래스(인프런, toycrane)는 양 끝에 고정한다.

## 배경

지금 `fetchCohorts`는 `name` 오름차순으로 정렬한다. 프로덕션 기준 결과는 `LG전자 1기 → 7기 → toycrane` 이고, 가장 오래된 기수가 맨 앞이다. 최신 기수를 쓰는 사람이 제일 많은데 목록 끝까지 내려가야 한다.

단순히 내림차순으로 뒤집으면 `T > L` 이라 **toycrane이 맨 위로 올라온다**. 기수가 아닌 항목이 최상단을 차지하는 게 이 작업의 출발점이었다.

## 결정

### 1. `cohorts`에 `display_order` 칼럼을 두되, 값은 자동으로 매긴다

`display_order int not null generated by default as identity`.

클래스를 추가하는 사람은 이 칼럼을 **적지 않는다**. 그냥 평소처럼 넣으면 된다:

```sql
insert into public.cohorts (name, label) values ('LGE-8', 'LG전자 8기');
```

넣은 순서가 곧 최신순이다. 나중에 넣은 행이 큰 값을 받고, 내림차순 정렬에서 위로 올라간다. 새 클래스는 거의 항상 가장 최신 클래스이므로 이게 원하는 결과다.

`by default`(`always` 아님)라서 필요하면 값을 직접 넘겨 덮어쓸 수 있다. 순서를 잘못 넣었거나 나중에 손봐야 할 때의 탈출구다.

### 2. 인프런과 toycrane 고정은 코드가 한다

정렬은 위 한 줄로 끝내고, 양 끝 고정은 이름 기반 규칙으로 `entities/cohort`에 둔다.

```
PINNED_TOP    = ["Inflearn"]    → 항상 맨 위
PINNED_BOTTOM = ["TOYCRANE"]    → 항상 맨 끝
그 외          → display_order 내림차순
```

이 코드베이스엔 이미 똑같은 모양의 규칙이 있다 — `entities/cohort/model/selectable.ts`의 `NON_SELECTABLE_COHORT_NAMES = ["TOYCRANE"]`. 예외를 이름으로 적어두는 방식이 낯설지 않고, 바로 옆에 붙는다.

**데이터가 아니라 코드에 두는 이유.** 고정을 숫자로 표현하면(인프런 100, toycrane 0) 그 숫자를 아는 사람만 클래스를 추가할 수 있다. 인프런의 100에는 아무 의미가 없어서 외우는 것 말고는 방법이 없다. 반면 예외는 두 개뿐이고 거의 바뀌지 않으므로, 코드에 적어두면 이름이 곧 설명이 되고 테스트로 고정된다.

### 3. `display_order desc, name asc`

동점일 때 순서가 흔들리지 않도록 `name`을 보조 키로 둔다. identity 칼럼이라 동점은 값을 직접 넘긴 경우에만 생긴다.

### 4. 세 화면에 모두 적용한다

`fetchCohorts` 하나를 고치면 보드 필터(칩 레일 + 콤보박스), 온보딩 클래스 선택, 설정 클래스 선택이 함께 바뀐다. 화면별로 다른 순서를 쓸 이유가 없다. 신규 가입자는 최신 기수일 확률이 높으니 온보딩에서도 최신순이 맞다.

### 5. `isSelectableCohort`는 그대로 둔다

toycrane을 온보딩·설정 선택지에서 빼는 규칙은 **선택** 가능 여부이고, `display_order`는 **표시** 순서다. 서로 다른 관심사이고 지금 분리돼 있는 상태가 맞다.

## 근거 — 다른 정렬 키를 쓰지 않는 이유

**`created_at`은 못 쓴다.** 프로덕션 값을 확인한 결과:

| name | created_at |
|------|-----------|
| LGE-2, LGE-3 | 2026-04-14 (동일 timestamp) |
| LGE-1 | 2026-05-11 |
| LGE-4 | 2026-05-21 |
| LGE-5 | 2026-05-26 |
| TOYCRANE | 2026-07-06 |
| LGE-6 | 2026-07-09 |
| LGE-7 | 2026-07-16 |

1기 행이 2·3기보다 **나중에** 채워졌다. 생성순으로 정렬하면 1기가 3기 위로 올라오고, 2기와 3기는 timestamp가 같아 순서가 비결정적이다. toycrane도 5기와 6기 사이에 끼어든다. 행이 만들어진 시각은 클래스가 열린 순서가 아니다.

**`name` 파싱도 못 쓴다.** [add-seed-sql/spec.md:56](../../../artifacts/add-seed-sql/spec.md)이 슬러그를 "테스트·URL·서버 조회가 쓰는 안정적 식별자"로 못 박았다. 표시 순서를 여기서 유추하면 그 계약을 넘어선다. 실무적으로도 `Inflearn`엔 숫자가 없어서 답이 안 나오고, `LGE-10`이 생기면 사전순으로 `LGE-1`과 `LGE-2` 사이에 낀다.

**`program` + `sequence`로 나누지 않는다.** 인프런이 기수별로 늘어난다면 프로그램 우선순위와 기수 번호를 분리하는 게 맞지만, 인프런은 기수 구분 없이 한 항목으로 들어온다. 칼럼 하나로 충분하다.

**프로젝트 활동량 순으로 정렬하지 않는다.** 프로젝트가 없는 신규 기수가 맨 아래로 가서 목적과 정반대가 된다.

## 선행 작업 — PR #180 (닫힘)

이 작업은 한 번 만들어졌다가 버려졌다. 브랜치 `cohort-filter-ui-0a7f76`, [PR #180](https://github.com/toy-crane/claude-hunt/pull/180), 커밋 `db35851` + `31abe8c`.

정렬 쿼리는 지금 우리가 정한 것과 **글자 그대로 같다** — `.order("display_order", { ascending: false }).order("name", { ascending: true })`. `created_at`을 쓸 수 없다는 판단도 같은 결론에 도달해 있다. 마이그레이션도 스키마 변경 + DML 백필 2개로 나뉘어 있어 구조가 그대로 쓸 만하다.

버려진 이유는 정렬이 아니다. PR이 **칩 행을 드롭다운으로 교체하는 UI 변경과 묶여 있었고**, main은 그 뒤 다른 방향(콤보박스 + 한 줄 레일 병행)으로 갔다. UI가 반려되면서 정렬까지 같이 사라졌다.

**가져올 것**: 정렬 쿼리, 마이그레이션 2단 구조, `fetch-cohorts.ts` 주석.
**바꿀 것**: 아래 두 가지.
**가져오지 말 것**: `3c364eb`, `4aa39ab` (UI 커밋 — 이미 대체됨).

### 그 구현과 달라지는 점

1. **`default 0` → identity.** PR #180은 `display_order int not null default 0`이었다. 클래스를 추가할 때 숫자를 매번 적어야 하고, 빠뜨리면 0이 들어가 조용히 맨 아래로 간다. 이게 이번에 지적받은 바로 그 문제다.
2. **인프런 고정이 없다.** 그 구현은 seed에서 인프런에 4를 줘서 우연히 맨 위에 있을 뿐이다. `LGE-5`가 들어오는 순간 5 > 4라서 인프런이 밀려난다. 고정이 아니라 그냥 숫자다.

`default 0`은 toycrane을 맨 끝에 두는 장치로도 쓰이고 있었는데(주석에 그렇게 적혀 있다), 이러면 값을 빠뜨린 새 클래스가 toycrane과 같은 자리를 두고 경쟁한다. 고정을 이름 규칙으로 옮기면 이 얽힘이 풀린다.

## 검증 결과 — 실제 데이터

프로덕션(8행)에 제안한 규칙을 그대로 돌린 결과:

```
LG전자 7기(0) → 6기(20) → 5기(22) → 4기(13) → 3기(20) → 2기(9) → 1기(3) → toycrane(3)
```

괄호는 프로젝트 수다. 목표한 순서와 정확히 일치한다. 정합성도 확인했다 — 전체 8행, `LGE-<n>` 패턴 7행, **규칙이 못 잡는 이름 0건**, 백필 값 중복 0건, 최댓값 7. 프로젝트 90개 전부 클래스에 연결돼 있고 고아 행이 없다.

인프런 고정은 로컬 DB(인프런 + `LGE-1`~`LGE-7` + toycrane, 9행)로 확인했다. `인프런 → 7기 → … → 1기 → toycrane` 순으로 양쪽 고정이 모두 걸린다.

**로컬 DB는 깨끗한 기준이 아니다.** PR #180의 마이그레이션이 공용 로컬 Supabase에 적용된 채로 남아 있어 `display_order` 칼럼이 이미 존재한다(`integer not null default 0`). 프로덕션에는 이 칼럼이 없다. 구현 전에 `supabase db reset`으로 이 브랜치 기준으로 되돌려야 한다.

## 성공 기준

- [ ] 로컬 seed에서 `fetchCohorts()`가 `인프런 → LG전자 3기 → 2기 → 1기 → toycrane` 순으로 반환한다
- [ ] 보드 콤보박스 목록이 `모든 클래스` 다음부터 같은 순서로 렌더된다
- [ ] 보드 칩 레일이 `모든 클래스` 다음부터 같은 순서로 렌더된다

- [ ] 설정 클래스 선택지가 같은 순서이고 toycrane이 없다
- [ ] **`display_order`를 적지 않고 새 클래스를 INSERT하면 목록 맨 위(인프런 바로 아래)에 나타난다** — 이 작업의 핵심 기준
- [ ] `Inflearn` 행을 나중에 INSERT해도 맨 위에 온다 (`display_order` 값과 무관하게)
- [ ] 프로덕션 마이그레이션 후 목록이 `LG전자 7기 → 6기 → … → 1기 → toycrane` 순이다

## 가정 — 다르면 알려주세요

- **"모든 클래스"는 계속 맨 앞 고정.** `display_order`의 영향을 받지 않는, 목록 밖의 항목이다.
- **프로젝트 0개인 클래스도 그대로 보여준다.** 최신인 7기가 지금 0개라 맨 앞 칩이 `LG전자 7기 0`이 된다. 숨기면 정작 자기 클래스를 찾는 7기 수강생에게 안 보이고, 첫 제출이 들어오는 순간 칩이 튀어나와 목록이 흔들린다. 카운트가 이미 빈 상태를 말해주고 있다.
- **기존 필터 동작은 건드리지 않는다.** URL 동기화, 칩 자동 스크롤, 콤보박스 검색은 그대로다.
- **클래스는 넣는 순서 = 최신 순서다.** 8기를 넣고 나서 9기를 넣는다. 이 전제가 깨지는 경우(누락된 옛 기수를 뒤늦게 채워 넣기)엔 값을 직접 넘겨 바로잡는다.

## 남은 위험

- **로컬 DB가 오염돼 있다.** PR #180의 마이그레이션이 공용 로컬 Supabase에 남아 `display_order`가 이미 있다. 이 상태로 `supabase db diff`를 돌리면 스키마 파일엔 없는 칼럼이라 **DROP이 생성될 수 있다**. 반드시 `supabase db reset` 후에 시작할 것.
- **identity 칼럼을 비어 있지 않은 테이블에 추가할 때 기존 행의 값.** Postgres가 기존 8개 행을 자동으로 채우지만 물리적 순서를 따르므로 기수 순서와 일치한다는 보장이 없다. 추가 후 `LGE-1`~`LGE-7`을 1~7로 명시적으로 UPDATE하고, identity 시퀀스를 `setval`로 최댓값 위까지 올려야 한다. 이걸 빠뜨리면 다음에 추가하는 클래스가 기존 값과 충돌한다. 마이그레이션 리뷰의 핵심 항목.
- **`fetch-cohorts.test.ts`의 목이 `.order()` 체이닝을 못 받는다.** 현재 `select: () => ({ order })`이고 `order`가 프로미스를 반환해서, `.order()`를 두 번 부르면 목이 깨진다. 체이닝 가능하도록 목을 고쳐야 한다.
- **프로덕션 백필과 배포 순서.** 칼럼이 생기기 전에 새 코드가 뜨면 `display_order` 정렬이 에러를 낸다. 마이그레이션 먼저, 배포 나중.
- **`seed.sql`의 낡은 주석.** 21–23행이 "fetchCohorts filters it out"이라고 적혀 있는데 커밋 51557ed 이후로 사실이 아니다. 같은 파일을 건드리는 김에 고칠 것.
- **인프런 고정은 프로덕션에서 검증할 수 없다.** 행이 없어서 로컬 seed로만 확인 가능하다. 실제로 들어올 때 맨 위에 오는지 다시 볼 것.
- **고정 목록이 코드에 있다는 건 인프런 추가가 코드 배포를 요구한다는 뜻은 아니다.** `Inflearn` 이름이 이미 목록에 있으므로 행만 INSERT하면 된다. 다만 **다른** 이름으로 들어오면(`inflearn`, `INFLEARN-1` 등) 고정이 안 걸린다. 슬러그를 `Inflearn`으로 맞출 것.
